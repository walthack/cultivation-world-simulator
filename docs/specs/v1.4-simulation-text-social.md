# v1.4 Spec — Scenario-Aware Simulation Text + Social Simulation

## Status
Draft — 2026-06-09 (Hassan). Prereq: v1.3 (Source Control ✅ / Narrative Injection partial)。

## Context（v1.3 closeout 暴露的三块）
1. **Narrative injection 弱**:prompt 通了但 LLM 没遵循 —— prompt hierarchy ≈ 90% 修仙 + 10% 六朝,注入点(NPC 目标)terms 0%。
2. **机械事件完全没覆盖**:sect_decider / 结算 / 突破 / 移动 / 招募 占 18 月 sweep 的 **51%**,全是修真术语,v1.3 一点没碰。
3. **关系不演化**:无自动社交 phase,50+ 月 sweep dialogues/relationships = 0。

## Goal
让剧本世界的**运行时文本(叙事 + 机械)整体吃 scenario context**,且**人物关系自动演化** —— 达到「长得像、读着像、人物会动」。
> 仍是 scenario-aware **generation / simulation**,**不碰剧情自动发展**(那是 v1.6-v1.9)。

## Scope (M1–M4)

### M1 — Mechanical Event Localization
机械事件(占 51%)文本 scenario 化。两类分治:
- **硬编码模板**(如「收入 X 灵石」「年度结算」):走 **term_map 术语映射层**(D1),渲染时替换术语。
- **LLM 生成**(如 `sect_decider` 的 sect 思考):接 narrative_context 注入(`sect_decider` 是 v1.3 漏掉的新注入点)。
- 对象:sect_decider / 年度结算 / 突破 / 移动 / 招募 / 修炼。
- 验收:机械事件文本不再纯修真术语。

### M2 — Narrative Injection 强化（修 v1.3 hierarchy）
解决「prompt reached ✅ / model obeyed ❌」:
- **scenario-aware world_lore**(D2):scenario 可提供自己的 world_lore 主体,**替换/前置**而非 append 末尾 —— 让世界设定本身是剧本的,不被修真 lore 淹没。
- **术语映射进 prompt**:prompt 内修真术语 → scenario 术语(与 M1 共享 term_map)。
- **prompt 权重/位置**:scenario context 前置 + 强指令(如「严格按该时代风格,避免修真宗门秘境术语」)。
- 验收:注入点(NPC 目标/事件叙事)terms 从 v1.3 的 0% 实质提升。

### M3 — Social Simulation Phase（原 v1.3 Phase 4 的实现）
- simulator lifecycle 新增**自动社交 phase**:同 region NPC 按 friendliness / 概率自动发起 `Conversation`(走现有 LLM 对话 + `RelationDeltaService`)。
- 频率受控(`generation_profile` / `game.story.probabilities`),避免淹没事件流。
- 关系随对话 / 共事演化。
- 验收:50+ 月 sweep `dialogues > 0` / `relationship changes > 0`;no-scenario 行为可配。

### M4 — 关系进 LLM 决策 context
- NPC 目标 / 对话 / 行为生成 prompt 注入该 NPC 的**关系网摘要**(friendliness / relations)。
- NPC 决策考虑关系(对敌对/盟友 NPC 的不同目标与反应)。
- 验收:NPC 行为/对话体现关系(可验证 case)。

## 设计决策

### D1 — `term_map`（术语映射,M1+M2 共享）
`generation_profile` 新增 `term_map`(修真术语 → scenario 术语)。应用于机械模板渲染(M1) + LLM prompt(M2)。
liuchao 例:`灵石→钱粮/资财`、`宗门→门阀/势力`、`突破→晋阶/进境`、`修为→功业`。
> 风险:映射覆盖度 —— 不全则残留修真术语。需列出高频术语清单优先覆盖。

### D2 — scenario world_lore 模式（M2）
`narrative_context` 加 `world_lore_mode: replace | prepend | append`(默认 `append` = v1.3 行为)。scenario 提供完整 world_lore 主体时用 `replace`,从根上扭转 prompt hierarchy。
> 风险:`replace` 要 scenario 写完整 lore,否则世界设定变空 —— liuchao 需补一份六朝 world_lore。

### D3 — Social phase（M3）
lifecycle 新 phase,经 `SimulationStepContext` 协调;频率配置;关系演化走 `RelationDeltaService`。属 simulator 架构改动,需充分回归(no-scenario 默认行为)。

## 验收（v1.4 RC target）
- 机械 + 叙事事件**整体 terms ratio 实质提升**(localization 后重定 —— v1.3 的 13% 受机械事件稀释,v1.4 后才有意义谈 ≥40%)。
- `dialogues > 0` / `relationship changes > 0`。
- NPC 决策体现关系(case)。
- **no-scenario zero regression**(每 M 配 negative test)。

## 实施顺序（codex,每 M 配 negative test + sweep 对比）
1. **M2**(修 hierarchy + term_map 进 prompt)—— v1.3 遗留最痛,先解决「读着像」。
2. **M1**(机械 localization,复用 M2 的 term_map)。
3. **M3**(social phase)—— simulator 架构改动,最重。
4. **M4**(关系进 LLM context)。

## 非目标
- 剧情自动发展(v1.6-v1.9 剧情阶梯)。
- 内容厚度 / Editor UI(v1.5)。
- 小说自动生成 scenario(v2)。

## 决策（御主 + codex 二审收敛,2026-06-09 拍板）
1. **term_map 覆盖** = **分层扩展表**:从 M1 事件模板 + M2 prompt 词汇**按实测频率**建 v1.4 词表,不做全量扫表。未命中术语留作可测量缺口,不 scope creep。(否决"拍脑袋选高频5词"——用已有事件模板反推覆盖面更精准。)
2. **world_lore 模式** = **`replace`**:liuchao 补一份完整六朝 world_lore,彻底切断旧修真 hierarchy。**`prepend` 被否** —— 它等于重做 v1.3 失败修法(六朝 lore 前置后 prompt 仍 90% 修仙、LLM 不遵循);不彻底切旧 lore 就没意义。
3. **Social phase** = **仅 scenario 开启**,no-scenario sandbox 零回归(架构契约,不在一个 milestone 同时移两个默认值)。scenario 数据好看后再以 opt-in 配置开放 sandbox。
4. **terms RC** = **按事件类型分阈值**:叙事类(测 LLM 遵从度)/ 机械类(测 term_map 确定性)各设独立阈值(两条生成路径、两类失败模式,聚合会互相掩盖,v1.3 的 13% 即教训)。**≥40% 整体仅作 release floor 兜底**,非主诊断指标。

## M2 Closeout（2026-06-10,定性达标）
- **达标(定性)**:world_lore `replace` + 强指令修了 v1.3 的 prompt hierarchy;整体 terms 13%→23-30%;LLM **能**产六朝政治叙事(sweep 实见「敛财聚势壮大兰亭清谈社」「教化关中」)。
- **term_map 边界教训(重要)**:`term_map` 只替换**注入到 prompt 的文本**(world_lore/context)+ **硬编码模板**,**不替换 LLM 生成的输出**。补的境界词(金丹/筑基/元婴/化神/证道/洞府/灵脉/飞升,共 18 词)对 LLM 注入点输出无效 → **归 M1 机械模板渲染用**(突破/结算等硬编码文本)。
- **注入点测量限制**:每 sweep 仅 ~4 个 NPC 目标事件,terms 在 0-33% 剧烈波动,样本不足以精确测注入点;以**定性 + 整体 ratio** 为准,不纠结注入点单点数字。
- **境界残留 → v1.5**:NPC 目标残留「突破金丹/元婴」是 **progression 机制层**(LLM 顺境界机制生成),term_map 和 prompt 都治不了;realm/progression 机制 scenario 化属 v1.5 content/mechanics layer,不在 v1.4。

## v1.4 最终验收报告（2026-06-10）

M1-M4 全交付,综合 sweep(liuchao 24 月)验收。

### 验收数据
| 维度 | v1.3 基线 | v1.4 |
|---|---|---|
| 整体 terms ratio | 13% | **50-59%** |
| dialogues | 0 | **5** |
| relationship changes | 0 | **8** |
| 全量 pytest | — | **1787 passed**(3 init sect-mock pre-existing) |

### 按 milestone
- **M1 机械 localization**:结算/突破/移动/招募走 term_map 六朝化(结算 100%、移动 57%);机械占 events 51%,这块六朝化把整体 terms 13%→59%。
- **M2 narrative 强化**:world_lore `replace` 修 v1.3 prompt hierarchy,注入点叙事六朝化(出现「敛财聚势,壮大兰亭清谈社」)。
- **M3 social phase**:自动社交 lifecycle phase(仅 scenario 开),dialogues 0→5、relationships 0→8 —— 人物会动。
- **M4 关系 context + Conversation narrative**:关系摘要进 NPC 目标/对话/行为 prompt;Conversation 注入六朝 context,对话古风化(「负手/拱手/师叔祖/药圃」,对比 M3 的「洞府/金精」)。

### 对照 RC target(决策 4:按类型分阈值,≥40% floor)
- ✅ 整体 terms **50% > 40% release floor**
- ✅ dialogues > 0 / relationship changes > 0(spec 原列为 v1.4 之外,M3 **超额达成**)
- ✅ no-scenario zero-regression(每 M negative test)
- ✅ 全量 pytest 通过(无新增失败)

### reframe 印证
对话仍保留仙门元素(吐纳/剑/灵株)+ 六朝政治古风(士族/门阀/师叔祖)—— 符合「六朝政治 + 仙门混合设定」,terms 50% 是该混合的**合理区间而非缺陷**。境界 progression 机制保留 → v1.5。

**v1.4 达标:「长得像、读着像、人物会动」的剧本世界落地。**
