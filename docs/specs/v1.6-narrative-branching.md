# v1.6 Spec — L1 剧情分支化（Narrative Branching）

## Status
Draft — 2026-06-11 (Hassan)。Prereq: v1.5(progression + content depth + cross-schema verification)✅ on main。
roadmap 定位:剧情能力阶梯 **L1**（③剧情自动发展的第一阶,仍 scripted,脱离单线）。

## Context — 现状与 delta

scenario engine 现有原语**已能表达原始分支**:
- **condition DSL**(`condition_evaluator.py`):`world_flag` / `var_equals` / `event_triggered` / `player_stat` / `random_chance` / `all|any|not` …
- **事件级编排**:`requires_events`(必须已触发)、`blocks_events`(触发后压制其它事件)。
- v1.5 实证:liuchao `han-song-aftermath-loyal` / `-hedge` 两事件分别 gate 在 `cheng_loyal_to_song` / `cheng_hedged_han` 上 —— **这已是一条手搓分支**。

> **所以 v1.6 不是「从零加分支」,而是把分支从「作者手搓 flag 簿记 + 逐变体事件 + 手动 blocks_events」升为「一等原语 + 引擎托管互斥 + 可验证可视」。**

现状手搓分支的痛点(= v1.6 要解决的):
1. **无一等「分支点」概念**:作者要为每条岔路写独立事件 + 各自 condition,N 路分支 = N 个事件 + 手动两两 `blocks_events`,易漏、易冲突。
2. **无「剧情线(storyline)」概念**:事件散落,无法表达「这批事件属于 A 线,选了 B 线后 A 线整体不再触发」。互斥靠逐事件 flag,规模一大不可维护。
3. **不可验证**:没有「分支必互斥、每条线可达、无悬空 goto」的结构校验(v1.5 我靠临时脚本查 flag 链,非引擎保证)。
4. **不可视**:作者看不到剧情树,无法确认分支覆盖。

## Goal
让作者用**一等原语**声明「按世界状态在此处岔向不同剧情线」,引擎**托管互斥与可达性**,并**可结构校验**。仍 100% scripted、确定性可玩可验 —— 为 v1.7(L2 LLM 填充叙事)备好骨架结构。
> **不碰 LLM 生成**(那是 v1.7+)。v1.6 是纯 scripted 的分支**结构层**。

## 设计选项（待御主拍 schema 方向）

### 选项 A — 分支点事件(branch-point),最轻
新增事件字段 `branches`:引擎按序评估各分支 condition,选**首个命中**,设一个 branch 选择器 var(如 `set_var: storyline=A`);后续事件用现有 `var_equals` gate。
- ✅ 改动最小(复用 condition + set_var),L1「工程可控、快速交付」最贴合。
- ⚠️ 互斥仍半手动(后续事件要自觉读 var);无强「剧情线」对象。

### 选项 B — 一等 storyline / arc 对象,最完整
事件加 `storyline: <id>`;新增 branch-point 选定 active storyline;引擎**只 dispatch active storyline 的事件**,其余整线自动压制。
- ✅ 心智模型最清晰(命名剧情线 + 引擎托管整线互斥);天然支持可视化(按 storyline 画树)。
- ⚠️ schema 改动较大(事件归属、storyline 注册表、active 集合状态);需迁移现有事件。

### 选项 C — 仅形式化现有 flag 模式,最保守
不加 schema,只加 helper + 结构校验(分支互斥、flag 链可达)+ 文档。
- ✅ 零 schema 风险。
- ❌ 没真正「脱离单线」,仍是手搓 —— **不满足 roadmap L1「一等分支」意图**,基本是 v1.5 的延续,不建议。

**Hassan 倾向**:**B 的「storyline 一等对象」是 L1 的正解**(roadmap 要的就是「按世界状态选不同剧情线」,storyline 是那个名词);但可**分两步落**:v1.6a 先做选项 A 的 branch-point + set_var(快速拿到分支能力)→ v1.6b 抽象出 storyline 对象 + 引擎托管整线互斥 + 校验。两步都可独立 ff-merge、各带负测试。

## 设计 Q（待御主拍）
1. **schema 方向** = A(轻/半手动互斥)还是 B(重/一等 storyline 引擎托管)还是 A→B 两步走?(建议 A→B 两步)
2. **互斥语义**:选定 B 线后,A 线事件是**永久压制**还是**可在条件满足时重新激活**(支持剧情线汇流/回流)?L1 建议先**永久压制(单向树)**,汇流留 L3。
3. **分支选择时机**:branch-point 在**事件 dispatch 时**评估(顺时序),还是支持**玩家显式选择**驱动分支(复用 `choices`→`set_var`)?建议两者都支持(condition 自动分支 + choice 显式分支)。
4. **校验严格度**:L1 是否强制「每条 storyline 可达 + 无悬空引用 + 分支互斥」为 load 时 hard error(同 v1.5 的 `_is_v02` 引用校验路子)?建议是 —— 结构正确性是 L1 的核心交付物。
5. **范围 demo**:用哪条线落地验证?建议把 v1.5 已有的 **han-song-loyal/-hedge 手搓分支重写为一等分支**(同时证明新原语能替代旧手搓,且不回归)。

## 提议 Milestones（待 Q 拍板后细化）
- **M1 — branch-point + set_var(选项 A 内核)**:事件支持 `branches`;引擎选首个命中分支并写 selector var;negative test(无命中分支 / 多命中取首 / 悬空 var)。
- **M2 — 结构校验**:load 时校验分支互斥 + 可达 + 无悬空引用(hard error)。
- **M3 — storyline 一等对象(选项 B,若拍 A→B)**:`storyline` 归属 + active 集合 + 引擎托管整线压制;迁移 han-song demo 为命名 storyline。
- **M4 — demo + 可视化(可选)**:剧情树 dump / 调试视图;把一条 liuchao 主线重写为多 storyline 分支验收。

## 验收（草案）
- 一条 liuchao 分支线用**一等原语**重写(替代 v1.5 手搓 flag 变体),行为等价或更优。
- 结构校验:故意写一条悬空/不可达/非互斥分支 → load **报错**(negative test 守门)。
- full pytest 零回归;no-scenario 路径不受影响。
- 跨 schema:branch 原语在 default/liuchao/sanguo 都能 load + 生效(延续 v1.5 `test_scenario_import_takes_effect` 思路)。
