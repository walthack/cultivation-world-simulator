# v1.5 Spec — Content Depth / World Polish

## Status
Draft — 2026-06-10 (Hassan). Prereq: v1.4 (Simulation Text + Social) ✅。

## Context
v1.0-v1.4 把 scenario engine **机制**建全了(导入/source/narrative/social),世界「长得像、读着像、人物会动」。但还**薄、难上手、体验糙**:
- 六朝内容**薄**:scripted 事件/角色/势力少(~10 events),sweep 里大量是通用修真模拟填充。
- **境界 progression 还是 default 修真**(v1.4 归来):realm/修炼体系没 scenario 化,NPC 目标还残留「突破金丹」。
- 玩家要**手写 JSON** 才能造 scenario(无 Editor)。
- 地图/势力/宗派**体验**未打磨。

## Goal
让六朝(及任意 scenario)世界**有厚度、好上手、体验完整** —— 从「机制能跑」到「内容丰满 + 玩家可造」。
> 仍**不碰剧情自动发展**(那是 v1.6-v1.9 剧情阶梯)。v1.5 是内容/机制/体验层,不是 narrative AI。

## Scope（候选,4 块,性质不同,需御主定优先级与拆分）
1. **六朝/三国 scripted 内容厚度**:扩 timeline 事件、历史角色、势力(目标向 80+ events,vs 现 ~10);让世界主线/支线丰满。
2. **境界 progression scenario 化**(v1.4 deferred):realm/修炼体系可被 scenario 替换为该世界的成长体系(如六朝的功业/官阶/名望/军功),或 scenario 可配 progression;根治「突破金丹」这类机制层残留。**这是 mechanics layer 改动,最重。**
3. **Scenario Editor UI**:玩家可视化造 scenario(角色/势力/地理/timeline),不手写 JSON;降低 mod 门槛。
4. **地图可视化 / 势力外交 / 宗派体验**打磨。

+ **backlog 清**:3 个 `test_game_init_integration` sect-mock failures(pre-existing 待清)、persona source 特判统一。

## 设计 Q（待御主拍）
1. **v1.5 优先级/拆分**:这 4 块一版做不完。先聚焦哪块?
   - 建议:**①内容厚度 + ②progression scenario 化**先做(直接提升"世界感"和"剧本贴合度");③Editor UI、④体验打磨可拆 v1.5b 或后续。
2. **progression scenario 化深度**:realm 机制**完全换**(六朝纯历史成长,无修真)?还是 **scenario 可配**(保留修真为 option,符合六朝"政治+仙门"混合)?—— 建议后者(与 v1.4 reframe 一致:六朝本就仙门并存)。
3. **内容厚度谁来写**:scripted 六朝事件/角色是内容创作(codex 写 + 御主/Hassan 审剧情贴合度)?还是等 v2 Novel-to-Scenario 半自动生成后再厚?
4. **backlog 3 个 init failures**:v1.5 顺手清,还是单独小修?

---

## 决策（御主 + codex 二审收敛,2026-06-10 拍板）
1. **scope** = **①内容厚度 + ②progression scenario 化(紧耦合)**;**③Scenario Editor UI / ④地图势力宗派体验 defer v1.6**。② progression schema **先定**(否则 scripted event 写死旧成长体系假设、后续返工),① 内容按新轴写。两者依赖强、交织做。
2. **progression 深度** = **scenario-configurable profile**(同一套 schema 支撑两种世界观,**不全换**):default=修真境界;liuchao 声明历史轴(功业/官阶/名望)+ 保留仙门轴并存。全换会砸既有修真 scenario 兼容 + 维护双 engine 分支,违 v1.4"政治+仙门"reframe。
3. **内容谁写** = **codex 写 scripted 六朝事件/角色 + 人审剧情贴合,不等 v2**。v2 Novel-to-Scenario 在 roadmap 末段,等它 = v1.6 branching/v1.7 LLM 生成无 authored bones 可依托、薄世界拖穿 v1.6-v1.8。人审是必要成本非 blocker。
4. **3 个 init sect-mock failures** = **独立小 fix 分支先清**(恢复全量绿基线),不混 v1.5(测试存量债、与 v1.5 无逻辑关联,混进会污染回归归因)。

**实施顺序**:① 独立清 init failures(干净基线)→ ② progression schema(profile 配置化)→ ③ 内容(按新轴批量写 scripted)。
