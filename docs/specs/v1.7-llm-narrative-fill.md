**本版仅设计草案 + Design Q —— 决策未拍、未实现、未 dispatch codex。**

## 边界（锁死,先于一切）

> **v1.7 = LLM Narrative Fill。不是 LLM 决定剧情走向。**

| 层 | 谁决定 | 备注 |
|---|---|---|
| 发生什么 / `effects` / `flags` / `branch`·`storyline` 选择 | **scripted event（确定性）** | v1.6 已建,保持不变 |
| event narration / dialogue flavor / scene description | **LLM（仅呈现层）** | v1.7 新增 |

**硬不变量(架构级,不可越界)**:
1. LLM 输出**只进入「展示文本」字段**(narration/dialogue/description),**绝不回流**到 effect / condition / branch / storyline 引擎。
2. 生成文本**不得改变任何机制结果** —— 同一事件,有无 LLM 填充、填充成功与否,其 effects/flags/branch outcome **逐字节一致**。
3. LLM 文本与 scripted 机制是**单向、隔离**的:机制 → 渲染请求 → LLM 文本 → 展示;反向无通路。

> 这条边界是 v1.7 的全部意义:让世界**读起来活了**,但走向**仍可验、可复现、不失控**。L3/L4(v1.8/v1.9)才让 LLM 碰走向。

### 架构决策(codex 二审后锁定 —— M1 之前必须先定)
**最关键决策:生成文本存放在哪。** codex 指出两个「显然」的位置都会泄漏到机制,已核实属实:
- ❌ **不能放进 `ScriptedScenarioState.state` / scenario_vars** —— `var_equals` 谓词直接读 `get_scenario_vars`(`condition_evaluator.py:152`),放这里 = LLM 文本可被 condition 读到,污染走向判定。
- ❌ **不能替换 `Event.content` / 复用 `Conversation` 输出** —— 事件/对话文本会经 `RelationDeltaService.resolve_event_text_delta`(`conversation.py:199`)驱动关系 friendliness 变化,并进入 AI 行动决策记忆;替换它 = fill 开/关两次运行的关系与后续行为**发散**,破坏「机制逐字节一致」。
- ✅ **专属只读渲染通道**:生成文本存入一个**独立的 render-only 字段/缓存**,**排除**于 `state`、`Event.content`、condition/effect 输入、AI 记忆之外。
- **守门方式**:**机制状态投影的差分测试** —— 同一时间线,fill 开 vs fill 关,**跨多个 tick** 的 effects/flags/branch/relations/AI-memory **逐字节一致**。这是 v1.7 的核心验收,必须在任何生成代码之前先立(见 M0)。

## Context / 阶梯定位

- roadmap **L2**(`docs/specs/roadmap-to-narrative-goal.md` line 40):timeline 事件是「骨架」(发生什么),LLM 据世界状态生成具体叙事/对话文本;走向仍 scripted,表现 LLM 化。
- **咬合 v1.6**:v1.6 给了骨架结构(branch/storyline,确定性选线);v1.7 给骨架的**表现层**。
- **复用现有基建,不从零造**:v1.3 narrative context injection、v1.4 scenario-aware world_lore + 术语映射(灵石→资财…)+ relationship context、progression context —— prompt 通路与 scenario-aware 上下文已在。v1.7 在其上加「按 scripted 事件生成**本事件**叙事文本」这一层。

## Goal
让标记需要丰富表现的 scripted 事件,在触发时由 LLM 据世界状态生成贴合的叙事/对话文本;失败可回退 authored 静态文本;生成结果可复现。机制层零改动、零风险。

## 设计 Q（待御主拍 —— 先列清,不实现)

### Q1 — 生成时机
事件触发时**同步**生成(阻塞该 tick 渲染),还是先用 **fallback text、后台异步**补?
- 同步:即时一致、实现简单;但 LLM 延迟会卡 tick / 拖慢体验。
- 异步:不卡 tick;但短暂显示 fallback,需回填 + 「文本已更新」通路。
- **Hassan 倾向**:M1 先**同步 + 强 fallback**(简单可验);异步留 M4 视延迟实测再上。
- **codex 反驳(采纳)**:`Simulator.step()` 在 scenario phase **await 之后**才跑后续 phase + finalization(`simulator.py:92`),单次 provider 请求可阻塞 **120s**(`client.py:299`),多个 fill 事件串行叠加 → 纯同步会卡死整 tick。**修正**:M1 不做「逐事件同步阻塞」,改为**单次有界批量调用**(一个月内所有 fill 事件合一次请求)**或 fallback-first(先回退、生成在 tick 外)**。「逐事件同步」方案否决。
- **codex 复核选择 → (b) 单次有界批量同步**:M1 更简单,且消解 Q11「per-tick 预算」在异步下的歧义(异步回填跑在 tick 边界外,预算难界定)。**条件**:给注入的 generator 包一个**远小于 120s 的严格超时**,别卡 tick 循环。

### Q2 — 失败策略
LLM 失败(超时/报错/无 key)时,是否回退 scripted **static description**?
- **Hassan 倾向**:**是,且 authored fallback 为必填** —— 每个可填充事件必须带静态 description 兜底,保证永不空白、永不卡死、no-LLM 环境可玩。

### Q3 — 可复现性
生成文本是否**持久化到 save**,避免 reload 后变?
- **Hassan 倾向**:**是** —— 生成一次即固化进 save / chronicle,reload 读缓存不重生成。否则破坏可复现(同一存档读出来剧情文字会变)+ 重复烧 token。键:`(scenario_id, event_id, [branch/choice outcome])`。
- **codex 反驳(部分采纳)**:持久化原则同意,但**存放位置不能在 `state`**(同架构决策,`var_equals` 可读)—— 需**独立 saved 字段**(save/load 现仅序列化 `state` + triggered ids,`save_game.py:245`/`load_game.py:125`,要新增)。且**键依赖 branch outcome,而 branch 选择结果现在没被持久化**(`branch_handler` 结果被丢弃;choice outcome 在 `event_outcomes` 但 branch 身份不可得)→ **前置依赖**:dispatch 须先把「解析出的 branch/choice outcome」暴露出来,才能构造稳定键。
- **codex 复核修正(Q3 vs Q8 一致性)**:键**必须含 `content_locale`** → `(scenario_id, event_id, resolved_outcome, content_locale)`,否则 zh-CN 与未来 locale 共用一槽互相覆盖。

### Q4 — 输入 context
给 LLM 哪些信息?候选:`world state` / `active storyline` / `involved avatars` / `recent chronicle` / `relationship graph`。
- 越多越贴合,但 token 成本与延迟越高。
- **Hassan 倾向**:**分层** —— 必给(event 骨架 name/description + involved avatars + active storyline + 已有 scenario-aware world_lore/术语);可配(recent chronicle 摘要 + relationship graph 子图,按事件需要)。给一个 context 预算上限。

### Q5 — 输出 schema
只要 `narration`,还是 `narration` + `dialogue` + `tags`?
- **Hassan 倾向**:**结构化但渐进** —— v1.7 起步 `{ narration }` 必备;`dialogue?: [...]`、`tags?: [...]` 作 option(M4)。结构化便于渲染分区 + 后续检索/审计。
- **codex 反驳(采纳)**:narration/dialogue 作有界字符串可以;但 **`tags` 移出 v1.7 scope** —— tags **不天然只读**,后续易变成检索/机制输入,违反「display-only」边界。v1.7 输出限定为命名类型里的有界 narration/dialogue **纯文本**,tags 不做。

### Q6 — 成本控制
**每个** scripted event 都生成,还是只给标了 `narrative_fill: true` 的事件生成?
- **Hassan 倾向**:**opt-in `narrative_fill: true`(per event,默认关)** —— 作者标记需要丰富表现的关键事件;纯机制事件(set_flag 之类)不烧 token。默认行为零成本、零风险。
- **codex 补充(采纳)**:dict-based loader 容忍新字段,加 flag 结构上容易;但 loader 现在不做类型强制(`"true"`/`1`/对象都收,`scenario_loader.py:832`)→ **要求显式 boolean,且在同一 loader pass 里 co-validate fallback 字段**(标了 fill 必须有 authored fallback)。

### Q7 — 安全边界(对应硬不变量 #2)
如何保证 LLM 文本**不得改 effects / flags / branch outcome**?
- **Hassan 倾向**:**架构隔离 + 守门测试** —— 生成发生在机制结算**之后**,LLM 只写 display 字段。
- **codex 反驳(采纳,关键)**:「生成在机制后 + 同 tick 状态快照不变」**不够** —— 若 fill 落进 `Event.content`,会经 AI 行动 prompt / 关系 prompt 影响**后续 tick**。守门测试必须:(a) 断言生成数据**不在 condition state、也不在 AI 记忆**;(b) **跨一个后续决策 tick** 验证,而非只验同 tick 快照。即 = 架构决策里的「跨多 tick 机制差分测试」。codex 注:`Simulator.step()` AI 决策在 phase 4、scenario 在 phase 12.5,故泄漏只会在**后续 tick** 影响机制 → 守门必须跨 tick。

### 补充 Design Q（codex 二审新增,原清单缺）
- **Q8 — Locale**:runtime 有持久化 `content_locale`(默认 zh-CN,5 个 locale,`save_game.py:137`)。生成用哪种语言?fallback 语言?cache 键是否含 locale?切换 locale 后已生成文本怎么办?
- **Q9 — 决定性 / cache-miss 恢复**:`call_llm_json` 不暴露 seed/temperature(`client.py:395`)→ 缓存未命中时重生成不可复现。Q3 管「持久化」,但**未命中后的恢复**(重生成 or 永久回退 fallback?)要定。
- **Q10 — Prompt 注入**:loader 对 `name`/`description` 无长度/字符约束(`scenario_loader.py:821`),LLM client 单 user-role prompt 无特权 system 消息(`client.py:271`)→ scenario 作者字段**未过滤**直灌 LLM。是否需净化/分隔 authored 内容与指令?
- **Q11 — 速率 / 并发预算**:dispatcher handler 串行(`event_dispatcher.py:49`),仅有全局 semaphore(`client.py:256`),**无 per-tick 调用/token 预算**。一个月多个 fill 事件的预算上限?(与 Q1 批量方案耦合)
- **Q12 — Chronicle 集成**:`EventStorage` 只有一个 `content` 字段 + 渲染元数据(`event_storage.py:81`)。需明确:**authored 文本 vs 生成展示文本**两者关系,哪个进 AI 记忆、哪个进日志/编年史。(直接关系到架构决策的隔离)
- **Q13 — Mock / fixture 策略**:`conftest.py` 只 mock 选定 manager,不 mock 任意 `call_llm_json`。fill 服务需**可注入的 generator callable** + 成功/超时/坏输出/未命中 的 fixtures,**测试不烧真 LLM 调用**。

## 提议 Milestones（待 Q 拍板后细化,tentative)
- **M0(gate,codex 新增)** — **render-only 通道 + 跨多 tick 机制差分测试,同一 milestone 内一起落**(fill 开 vs 关,effects/flags/branch/relations/AI-memory 逐字节一致)。**这是闸门:M1 起任何生成代码必须先过 M0。** 对应「最关键决策:生成文本存哪」。
  - **codex 复核修正**:差分测试**不能先于通道存在** —— 没有独立通道时 fill「开」无处可写,测试断言「生成数据不在 state/AI记忆」会**空过(vacuously green)**。所以通道(贯穿 `Event`/序列化/前端 DTO mapper/`useEventPanel`)与差分测试**必须同 milestone**,通道先落、测试才能真正在回归时变红。
- **M1** — `narrative_fill` schema(事件 opt-in 显式 bool + co-validate fallback)+ **有界批量/fallback-first 生成(非逐事件同步阻塞)**+ 可注入 mock generator + scripted fallback。
- **M2** — 独立 save/load 缓存字段(**非 `state`**)+ dispatch 暴露「解析出的 branch/choice outcome」以构造稳定键(生成一次固化,reload 复现)。
- **M3** — 有界 context 组装(world state / storyline / involved avatars / relationship 分层注入,**chronicle 输入仅限 authored/机制文本**,杜绝把已生成 fill 回喂成 context 的反馈环)+ token 预算。
- **M4**(option)— dialogue 结构化;异步 backfill(**仅在先定义好「事件文本更新投递 + 取消」机制后**)。tags 不在 v1.7 scope。

## 验收（草案）
- 标 `narrative_fill` 的事件触发后,展示文本为 LLM 生成;**失败回退 authored static**,不崩、不空白。
- **机制状态(effects/flags/branch)生成前后逐字节不变**(守门测试,兑现硬不变量)。
- reload save 后同一事件叙事文本不变(持久化)。
- **未标 `narrative_fill` 的事件零 LLM 调用**(成本守门)。
- no-scenario / 无 LLM key 时全链路走 fallback,正常可玩。
