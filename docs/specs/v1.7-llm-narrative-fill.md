# v1.7 Spec — L2 剧情细节 LLM 填充（LLM Narrative Fill）

## Status
Draft — 2026-06-13 (Hassan)。Prereq: v1.6（L1 branching）✅ on main。
**本版仅设计草案 + Design Q —— 决策未拍、未实现、未 dispatch codex。**

---

## 边界（锁死,先于一切）

> **v1.7 = LLM Narrative Fill。不是 LLM 决定剧情走向。**

| 层 | 谁决定 | 备注 |
|---|---|---|
| 发生什么 / `effects` / `flags` / `branch`·`storyline` 选择 | **scripted event（确定性）** | v1.6 已建,保持不变 |
| event narration / dialogue flavor / scene description | **LLM（仅呈现层）** | v1.7 新增 |

**硬不变量(架构级,不可越界)**:
1. LLM 输出**只进入「展示文本」字段**(narration/dialogue/description),**绝不回流**到 effect / condition / branch / storyline 引擎。
2. 生成文本**不得改变任何机制结果** —— 同一事件,有无 LLM 填充、填充成功与否,其 effects/flags/branch outcome **逐字节一致**。
3. LLM 文本与 scripted 机制是**单向、隔离**的:机制 → 渲染请求 → LLM 文本 → 展示;反向无通路。

> 这条边界是 v1.7 的全部意义:让世界**读起来活了**,但走向**仍可验、可复现、不失控**。L3/L4(v1.8/v1.9)才让 LLM 碰走向。

---

## Context / 阶梯定位

- roadmap **L2**(`docs/specs/roadmap-to-narrative-goal.md` line 40):timeline 事件是「骨架」(发生什么),LLM 据世界状态生成具体叙事/对话文本;走向仍 scripted,表现 LLM 化。
- **咬合 v1.6**:v1.6 给了骨架结构(branch/storyline,确定性选线);v1.7 给骨架的**表现层**。
- **复用现有基建,不从零造**:v1.3 narrative context injection、v1.4 scenario-aware world_lore + 术语映射(灵石→资财…)+ relationship context、progression context —— prompt 通路与 scenario-aware 上下文已在。v1.7 在其上加「按 scripted 事件生成**本事件**叙事文本」这一层。

## Goal
让标记需要丰富表现的 scripted 事件,在触发时由 LLM 据世界状态生成贴合的叙事/对话文本;失败可回退 authored 静态文本;生成结果可复现。机制层零改动、零风险。

## 设计 Q（待御主拍 —— 先列清,不实现)

### Q1 — 生成时机
事件触发时**同步**生成(阻塞该 tick 渲染),还是先用 **fallback text、后台异步**补?
- 同步:即时一致、实现简单;但 LLM 延迟会卡 tick / 拖慢体验。
- 异步:不卡 tick;但短暂显示 fallback,需回填 + 「文本已更新」通路。
- **Hassan 倾向**:M1 先**同步 + 强 fallback**(简单可验);异步留 M4 视延迟实测再上。

### Q2 — 失败策略
LLM 失败(超时/报错/无 key)时,是否回退 scripted **static description**?
- **Hassan 倾向**:**是,且 authored fallback 为必填** —— 每个可填充事件必须带静态 description 兜底,保证永不空白、永不卡死、no-LLM 环境可玩。

### Q3 — 可复现性
生成文本是否**持久化到 save**,避免 reload 后变?
- **Hassan 倾向**:**是** —— 生成一次即固化进 save / chronicle,reload 读缓存不重生成。否则破坏可复现(同一存档读出来剧情文字会变)+ 重复烧 token。键:`(scenario_id, event_id, [branch/choice outcome])`。

### Q4 — 输入 context
给 LLM 哪些信息?候选:`world state` / `active storyline` / `involved avatars` / `recent chronicle` / `relationship graph`。
- 越多越贴合,但 token 成本与延迟越高。
- **Hassan 倾向**:**分层** —— 必给(event 骨架 name/description + involved avatars + active storyline + 已有 scenario-aware world_lore/术语);可配(recent chronicle 摘要 + relationship graph 子图,按事件需要)。给一个 context 预算上限。

### Q5 — 输出 schema
只要 `narration`,还是 `narration` + `dialogue` + `tags`?
- **Hassan 倾向**:**结构化但渐进** —— v1.7 起步 `{ narration }` 必备;`dialogue?: [...]`、`tags?: [...]` 作 option(M4)。结构化便于渲染分区 + 后续检索/审计。

### Q6 — 成本控制
**每个** scripted event 都生成,还是只给标了 `narrative_fill: true` 的事件生成?
- **Hassan 倾向**:**opt-in `narrative_fill: true`(per event,默认关)** —— 作者标记需要丰富表现的关键事件;纯机制事件(set_flag 之类)不烧 token。默认行为零成本、零风险。

### Q7 — 安全边界(对应硬不变量 #2)
如何保证 LLM 文本**不得改 effects / flags / branch outcome**?
- **Hassan 倾向**:**架构隔离 + 守门测试** —— 生成发生在机制结算**之后**,LLM 只写 display 字段;加测试断言「同一事件,LLM 文本注入前后,effects/flags/branch outcome 完全不变」+「LLM 输出字段不在机制可读范围内」。

## 提议 Milestones（待 Q 拍板后细化,tentative)
- **M1** — `narrative_fill` schema(事件 opt-in 标记 + 输出字段)+ 同步生成 + scripted fallback + **安全边界守门测试**(机制前后不变)。
- **M2** — 持久化到 save(生成一次固化,reload 复现;按 Q3 的键)。
- **M3** — context 组装(world state / storyline / involved avatars / relationship / chronicle 分层注入,带 token 预算)。
- **M4**(option)— dialogue + tags 结构化输出;异步生成(若 Q1 选同步且延迟实测成问题)。

## 验收（草案）
- 标 `narrative_fill` 的事件触发后,展示文本为 LLM 生成;**失败回退 authored static**,不崩、不空白。
- **机制状态(effects/flags/branch)生成前后逐字节不变**(守门测试,兑现硬不变量)。
- reload save 后同一事件叙事文本不变(持久化)。
- **未标 `narrative_fill` 的事件零 LLM 调用**(成本守门)。
- no-scenario / 无 LLM key 时全链路走 fallback,正常可玩。
